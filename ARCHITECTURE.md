## 一、★ 核心研究框架:制度光谱

**研究问题**:
> 一个法案的命运,多大程度由它的**内容**决定,多大程度由它所处的**制度**决定?

三个法域恰好构成一条**制度约束由弱到强的光谱**——这不是硬凑,是真实的对照设计。

| | 法域 | 制度特征 | 负样本 | 系统角色 |
|---|---|---|---|---|
| **弱约束** | **加州** | 两院制 + 党派竞争,大量法案死于委员会或被否决 | ✅ 充足 | `prediction` 预测 |
| **中约束** | **香港** | 行政主导,政府草案通过率偏高,但仍有撤回/失效 | ⚠️ 待验证 | `retrieval` 检索(暂) |
| **强约束** | **澳门** | 《基本法》第75条:涉及政府政策的议案,**提出前**须获行政长官书面同意 | ❌ 几乎没有 | `control` 对照组 |

### 由此得出的核心洞察

> **当制度筛选发生在提案【之前】,"预测通过率"这个问题本身就消失了——
> 因为失败的提案从未被提出。**

**澳门无法预测,不是数据缺失,是制度使然。** 这个"发现问题本身不成立"的结论,
比"做了个准确率 85% 的模型"更有科学价值,也更诚实。

> 香港落在光谱的哪一档,由 `verify_hk_data.py` 的验证结果决定:
> 失败样本 >15% → 升为 `prediction`;<5% → 保持 `retrieval`。

## 二、v2 相比 v1 改了什么

| 维度 | v1 | v2 | 好 / 坏 |
|---|---|---|---|
| **法域** | 仅加州 | 加州 + 香港 + 澳门 | ✅ 构成制度光谱,有内在学术逻辑<br>✅ 中国元素**天然在内**,非硬贴 |
| **法域角色** | 无区分 | prediction / retrieval / control | ✅ 诚实:不硬把不可预测的说成能预测 |
| **语言** | 无 | 每法域独立语言矩阵 | ✅ 解决"多语社区看不懂法案"的真实痛点 |
| **模块划分** | 六个都叫 Agent | 3 子 Agent + 3 工具 | ✅ 更稳、好调试、少踩多智能体的坑 |
| **底层 Skill** | 一开始定义 | **推迟到尾声再写** | ✅ Skill 是经验沉淀,不是凭空规范 |
| **显示控制** | 无 | 预设切换(数据照跑,界面可切) | ✅ 一份项目,按受众调整展示重点 |
| **博彩模块** | 有(预测市场) | **已删除** | ✅ 政治事件博彩有法律与伦理风险 |
| **技术措辞** | "SGLang distillation" | "SGLang serving" | ✅ 没做蒸馏就别写蒸馏 |

**v2 的核心改进:诚实。** 不把没验证的写成已完成,不把不可行的说成可行。

## 三、三层架构

```
┌──────────────────────────────────────────────┐
│  顶层:Leges 主 Agent(大脑)                   │  ← 参赛主体
│  理解请求 → 选法域 → 编排调用 → 多语言输出      │
└──────────────────────────────────────────────┘
              │
   ┌──────────┼───────────────┬──────────────┐
   ▼          ▼               ▼              ▼
┌────────┐ ┌──────────┐  ┌──────────┐  ┌──────────┐
│ 子Agent │ │  工具     │  │ 法域适配层│  │ 语言层    │
│(会判断) │ │(确定功能)│  │(CA/HK/MO)│  │(多语输出) │
└────────┘ └──────────┘  └──────────┘  └──────────┘
```

**判断标准**:需要"动脑判断" → 子 Agent;确定的输入→输出 → 工具。

### 子 Agent(`engine/`)

| 模块 | 为什么是 Agent |
|---|---|
| **预测分析 Agent** ★ | 判断 + 归因(内容 vs 政治)+ 解释。**Amanda 研究核心** |
| **创作 Agent** | 决定风格、生成多版本倡导内容(多语言) |
| **社媒/反馈 Agent** | 分析公众反馈、决定如何迭代 |

### 工具(`server/`,MCP)

| 模块 | 为什么是工具 |
|---|---|
| **搜索工具** | 输入提案 → 输出相似法案(跨法域检索) |
| **图表工具** | 输入数据 → 输出可视化 |
| **文档工具** | 套模板 → 输出 PPT / 报告(多语言) |

## 四、语言层

### 关键认知:语言分两层

| 层 | 说明 |
|---|---|
| **数据层(原文)** | 加州 = 仅英文;香港 = 中英双语;澳门 = 中文 + 葡语 |
| **输出层(翻译)** | 把分析结果 / 倡导内容**翻译成用户选的语言** |

### 语言矩阵

| 法域 | 界面/输出语言 | 原文语言 |
|---|---|---|
| **加州** | English · 简体 · 繁體 · **Español** · **Filipino** | English |
| **香港** | English · 简体 · 繁體 | English · 繁體 |
| **澳门** | English · 简体 · 繁體 · **Português** | 繁體 · Português |

**为什么加州要西语/菲语**:加州有大量西语与菲语人口。
**加州不存在西语/菲语版的法案原文**——这两种语言的作用是**把倡导内容翻译给这些社区看**,
让他们看懂、愿意签名支持。提案需要公众签名,而人们只会签自己看得懂的东西。
*(状态:es/fil 翻译尚未实现)*

## 五、显示预设(数据照跑,界面可切)

**后端始终处理全部法域**;预设只控制**前端显示什么**。

| 预设 | 显示 | 默认语言 | 用途 |
|---|---|---|---|
| `hackathon` | HK · MO | 简体中文 | DGX Spark 黑客松 |
| `california` | CA | English | 加州立法课程 |
| `china` | HK · MO | 简体中文 | 展示港澳法域 |

切换方式——改 `engine/config.py` 一行:

```python
ACTIVE_PRESET = "hackathon"   # → "california" 或 "china"
```

或运行时:`get_active_config(preset="california")`

## 六、底层 Skill:**推迟到尾声再写**

Skill 的本质是"**把已摸索出的最佳实践固化**"。现在还没跑通模型、没生成过倡导内容——
此时写规范 = 凭空想象一件还没做过的事,大概率跟实际做出来的对不上,最后还要推翻重写。

**正确顺序**:先做出来(哪怕粗糙)→ 发现该怎么做才对 → 最后固化成 Skill。
**Skill 是沉淀,不是起点。**

唯一例外:预测特征 + 防泄漏规则,建模时**顺手记草稿笔记**(关系学术严谨性),
最后整理成 Skill。

## 九、诚实边界(必须守)

1. **不把没验证的写成已完成**:香港未验证前,标 `retrieval` 而非 `prediction`。
2. **不把不可行的说成可行**:澳门 `prediction_feasible=False`,并写明制度原因。
3. **没做蒸馏就别写蒸馏**:技术栈如实写 "SGLang serving"。
4. **倡导内容只生成原创**,绝不搬运有版权的视频/音乐。
5. **Amanda 研究独立性**:Leges 整体是团队作品(Hackathon 允许团队);
   但她的研究(内容 vs 政治归因、制度光谱分析)必须**她独立设计、分析、撰写**。
   工具和数据可共享,**研究的智力核心必须本人完成**。
